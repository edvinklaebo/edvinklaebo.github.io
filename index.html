About the Game

Three Choices is a tactical turn-based RPG built entirely in Unity. Every encounter challenges the player with meaningful decisions — each turn offers exactly three actions, each with cascading consequences. The game features a modular, data-driven architecture that makes it easy to extend enemy types, spells, and loot without touching game logic code.

Enemies are defined through event-driven data definitions, a layered status effect system handles buffs and debuffs at runtime, and combat events flow through a decoupled pipeline that keeps the UI, audio, and game logic entirely independent.

Under the Hood

Technical Breakdown

A look at the core systems powering Three Choices — each built to be modular, extensible, and data-driven.

🧩

Modular Enemy Definitions

Each enemy is defined as a data asset containing stats, skill sets, and loot profiles. Adding a new enemy type requires zero code — designers edit assets directly in the Unity Inspector.

EnemyDefinitionSO SkillSetSO
💫

Status Effect System

A layered effect stack applies buffs and debuffs at runtime. Effects are stackable, time-limited or turn-limited, and communicate through the combat event bus so the UI always reflects the current state without direct coupling.

StatusEffect EffectStack IEffectApplicable
🎲

Weighted Loot Tables

Drop chances are defined per-enemy as weighted probability tables. Weights are normalised at runtime so designers can tweak values freely without needing to ensure they sum to 100. Supports conditional modifiers (e.g. luck stat, difficulty scaling).

LootTableSO WeightedEntry<T> LootResolver
⚡

Event-Driven Combat Pipeline

All combat interactions flow through a central event bus. Actions raise typed events (OnDamageDealt, OnStatusApplied, etc.) that UI panels, the audio manager, and the replay recorder subscribe to independently — keeping each system fully decoupled.

CombatEventBus ICombatListener ActionResult
